47 /** A multi-dimensional domain over which to iterate. Used when
48  * defining functions as reductions.
49  *
50  * A reduction is a function with a two-part definition. It has an
51  * initial value, which looks much like a pure function, and an update
52  * rule, which refers to some RDom. Evaluating such a function first
53  * initializes it over the required domain (which is inferred based on
54  * usage), and then runs update rule for all points in the RDom. For
55  * example:
56  *
57  \code
58  Func f;
59  Var x;
60  RDom r(0, 10);
61  f(x) = x; // the initial value
62  f(r) = f(r) * 2;
63  Image<int> result = f.realize(10);
64  \endcode
65  *
66  * This function creates a single-dimensional buffer of size 10, in
67  * which element x contains the value x*2. Internally, first the
68  * initialization rule fills in x at every site, and then the update
69  * rule doubles every site.
70  *
71  * One use of reductions is to build a function recursively (pure
72  * functions in halide cannot be recursive). For example, this
73  * function fills in an array with the first 20 fibonacci numbers:
74  *
75  \code
76  Func f;
77  Var x;
78  RDom r(2, 18);
79  f(x) = 1;
80  f(r) = f(r-1) + f(r-2);
81  \endcode
82  *
83  * Another use of reductions is to perform scattering operations, as
84  * unlike a pure function declaration, the left-hand-side of a
85  * reduction definition may contain general expressions:
86  *
87  \code
88  ImageParam input(UInt(8), 2);
89  Func histogram;
90  Var x;
91  RDom r(input); // Iterate over all pixels in the input
92  histogram(x) = 0;
93  histogram(input(r.x, r.y)) = histogram(input(r.x, r.y)) + 1;
94  \endcode
95  *
96  * A reduction definition may also be multi-dimensional. This example
97  * computes a summed-area table by first summing horizontally and then
98  * vertically:
99  *
100  \code
101  ImageParam input(Float(32), 2);
102  Func sum_x, sum_y;
103  Var x, y;
104  RDom r(input);
105  sum_x(x, y) = input(x, y);
106  sum_x(r.x, r.y) = sum_x(r.x, r.y) + sum_x(r.x-1, r.y);
107  sum_y(x, y) = sum_x(x, y);
108  sum_y(r.x, r.y) = sum_y(r.x, r.y) + sum_y(r.x, r.y-1);
109  \endcode
110  *
111  * You can also mix pure dimensions with reduction variables. In the
112  * previous example, note that there's no need for the y coordinate in
113  * sum_x to be traversed serially. The sum within each row is entirely
114  * independent. The rows could be computed in parallel, or in a
115  * different order, without changing the meaning. Therefore, we can
116  * instead write this definition as follows:
117  *
118  \code
119  ImageParam input(Float(32), 2);
120  Func sum_x, sum_y;
121  Var x, y;
122  RDom r(input);
123  sum_x(x, y) = input(x, y);
124  sum_x(r.x, y) = sum_x(r.x, y) + sum_x(r.x-1, y);
125  sum_y(x, y) = sum_x(x, y);
126  sum_y(x, r.y) = sum_y(x, r.y) + sum_y(x, r.y-1);
127  \endcode
128  *
129  * This lets us schedule it more flexibly. You can now parallelize the
130  * update step of sum_x over y by calling:
131  \code
132  sum_x.update().parallel(y).
133  \endcode
134  *
135  * Note that calling sum_x.parallel(y) only parallelizes the
136  * initialization step, and not the update step! Scheduling the update
137  * step of a reduction must be done using the handle returned by
138  * \ref Func::update(). This code parallelizes both the initialization
139  * step and the update step:
140  *
141  \code
142  sum_x.parallel(y);
143  sum_x.update().parallel(y);
144  \endcode
145  *
146  * When you mix reduction variables and pure dimensions, the reduction
147  * domain is traversed outermost. That is, for each point in the
148  * reduction domain, the inferred pure domain is traversed in its
149  * entirety. For the above example, this means that sum_x walks down
150  * the columns, and sum_y walks along the rows. This may not be
151  * cache-coherent, but you may not reorder these dimensions using the
152  * schedule, as it risks changing the meaning of your function (even
153  * though it doesn't in this case). The solution lies in clever
154  * scheduling. If we say:
155  *
156  \code
157  sum_x.compute_at(sum_y, r.y);
158  \endcode
159  *
160  * Then the sum in x is computed only as necessary for each scanline
161  * of the sum in y. This not only results in sum_x walking along the
162  * rows, it also improves the locality of the entire pipeline.
163  */
